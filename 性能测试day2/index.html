关于性能测试

简单理解：对于性能指标的测试就是性能测试

功能测试关注能不能用，性能测试关注好不好用

性能测试开始时点：功能测试基本完成之后

性能关注点：

  • 接口响应时间：50毫秒~1000毫秒
  • 吞吐量：1000万每天，2000万每天
  • TPS：transaction per second 事务处理能力，每秒处理事务数（登录，打开页面，选择商品，加入购物车，下单，付款）
  • 备注：“日活” 每日活跃用户数，是运营数据，跟性能没有必然关联性

性能期望值的定义来源：

  • 竞品性能数据参考
  • 运营数据推算

性能测试关键技术：压力测试

  • 压力从何而来-压测工具模拟
  • 压力类型：
    • 短时大并发- 验证最大处理能力
    • 长时小并发- 验证长期运行稳定

影响系统性能的因素：代码性能，系统架构，网络架构，存储读写性能，硬件性能等

性能测试的价值：预测系统的性能瓶颈点，预测系统性能冗余度，平衡成本与收益

关于jmeter

JMeter简单介绍

主要功能模块：

  • 取样器：进行脚本逻辑控制
  • 线程组：压测场景控制
  • 监视器：监控压测运行，获取性能指标

常规菜单功能：

新建，保存，查询，配置，帮助等

JMeter使用说明

压测需求：http://news.baidu.com

压测页面：首页，国际频道，财经频道

操作步骤：

  1. 录制回放：
    1. 脚本录制之前的业务步骤梳理，明确
    2. 创建代理 HTTPS test script recorder
    3. 添加 Thread group
    4. 添加 recorder controller
    5. 代理中的配置：端口，include，exclude，存放位置
    6. 开启jmeter代理
    7. 配置浏览器代理
    8. 在浏览器完成业务操作
    9. jmeter录制完成后检查
  2. 回放验证
    1. 添加view result tree
    2. view result tree用途：看request和response详情
    3. 只是为了检验，所以一个并发一次循环即可，单击开始运行按钮
  3. 添加响应断言
    1. 检验请求、响应的正确性（校验点）
    2. 压测结果的有效性 - 请求收到正确响应的前提下
    3. 断言的实际值与设定值相符–>成功
    4. 断言的实际值与设定值不符–>失败
    5. 响应断言：校验响应码和文本内容
  4. 添加聚合测试报告 aggregate report
    1. 聚合报告：比较常用的一套展示压测结果的报告
    2. 展示请求与响应性能参数
    3. 累计值压测数据值的展示
    4. 建议连续压测，如果中间有间断需要数据清零

jmeter性能指标解读（聚合报告）：

  • Label,线程组的步骤名

  • # 样本,一共发出的请求数

  • Average 平均值,平均响应时间，默认情况下是单个请求的平均响应时间（ms）

    • 平均值的困扰 ~ 个别极端结果对实际性能数据的干扰
    • % Line 更能客观反映实际性能情形

  • Median 中位数,有一半的服务器响应时间低于该值而另一半高于该值，也就是50%用户的响应时间

  • 90% Line 百分位,90%请求的响应时间

  • 95% Line 百分位,95%请求的响应时间

  • 99% Line 百分位,99%请求的响应时间

  • Min 最小值,服务器响应的最短时间

  • Max 最大值,服务器响应的最长时间

  • Error 异常 %, 错误率

    • 测试出现的错误请求数量百分比
    • 确认是否允许的发生或者错误率允许在多大的范围内
    • 若出现错误就要看服务端的日志，配合开发查找定位原因

  • Throughput 吞吐量,tps

    • 默认情况下表示每秒处理的请求数，也就是指服务器处理能力，tps越高说明服务器处理能力越好
    • 吞吐量默认以request/second来衡量，即每秒多少个请求

  • Received 接收 KB/sec,每秒从服务器端接收到的数据量

  • Sent 发送 KB/sec,每秒从客户端发送的请求的数据量

JMeter 静默压测

  • 静默(脱离UI界面运行JMeter压测)
  • 好处：命令运行更容易搞事情
  • 命令格式： jmeter -n -t $jmx_file -l $jtl_file
  • jmx文件是程序脚本文件，压测控制过程记录在jmx文件中
  • jtl文件是压测请求响应数据的原始文件

关于接口

Restful API接口简介

  • 基于协议http或者HTTPS
  • url定位资源位置
  • 常见method：get，post，put，delete
  • 常见执行结果状态返回码
    • 200 - 成功
    • 302 - 重定向
    • 401 - 未授权
    • 404 - 资源不存在
    • 500 - 服务器内部错误
  • 直观上看就是url配合请求数据包，得到响应数据包

为什么要进行接口测试

  • 互联网项目的标准架构 - 前后端分离
  • 重后端，轻前端，逻辑的重头都在后端，后端的相对压力更大，一般影响系统性能的因素在后端
  • 后端通过Restful API接口与前端交互，所以对Restful API接口进行压测可以直观了解到系统的性能

练习：https://github.com/jesseATgithub/iJmeter

数据传递

添加调试样本 Debug Sampler

  • JMeter Properties jmeter属性
  • JMeter Variables jmeter变量
  • System Properties 系统属性
  • 一般只显示JMeter Variables就够用了

解析JSON响应数据

  • 需要使用json路径表达式来匹配要提取的变量值
  • $. 代表JSON的根节点
  • 可以在查看结果树的JSON Path Tester来确定所匹配的JSON变量路径的正确性
  • 在JSON提取器中填入如下信息
    • 变量名称
    • JSON路径表达式
    • 匹配编号（0表示随机）
  • JMeter中引用变量的方式 ${变量名}
  • 数据填写后运行调试结果

全局参数设置与CSV数据导入

  • 设定全局参数 User Defined Variables
    • 大家都要引用的参数，使用原则：有共性
    • 为了方便修改参数，一处改动全局替换
    • 常见的可以将hostname，IP，端口号，协议等定义为全局变量
  • 设定CSV数据导入 CSV Data Set Config
    • 多用户参数可以提取到CSV文件中
    • 配置数据也可以放到CSV文件中导入
    • 用excel创建CSV即可，数据之间使用英文的逗号来分割，保存后缀为csv文件

压测执行自动化

  • 脚本中添加聚合报告
  • 制定压测策略 不同的并发数 10 20 30 40 50
  • 记录结果
  • 测试期望结果
    • 验证能够支撑多大的并发数，峰值数
    • 验证错误率，定义可接受范围 <=0.1% or <=0.5% or must = 0%
  • 寻找系统性能点
